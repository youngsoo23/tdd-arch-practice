# 테스트 가이드: 범위, 대상, 그리고 팁

이 문서는 이 프로젝트를 진행하면서 정리한 테스트 관련 메모를 문서화한 것입니다. "얼마나 테스트해야 하는가"부터 "무엇을 테스트해야 하는가", 그리고 실전에서 쓸 수 있는 팁까지를 다룹니다.

## 1. 테스트는 왜 중요한가

테스트를 짜는 목적은 "커버리지를 채우는 것"이 아니라 **배포하는 개발자가 확신을 가질 수 있게 하는 것**입니다.

- 테스트가 없으면 배포 전에 "이 변경이 다른 곳을 깨뜨리지 않았을까?"라는 질문에 사람이 코드를 눈으로 읽으며 답해야 합니다. 코드베이스가 커질수록 이 방식은 느려지고 틀리기 쉬워집니다.
- 테스트가 있으면 그 질문에 기계가 몇 초~몇 분 안에 답해줍니다. 이 "안정적이라는 확신"이 리팩터링, 기능 추가, 의존성 업그레이드 같은 변경을 두려움 없이 하게 만드는 진짜 이득입니다.
- 테스트는 실행 가능한 문서이기도 합니다. `given/when/then`으로 구조화되고 한글로 상황을 서술한 테스트명은 "이 도메인 규칙이 무엇인가"를 코드보다 먼저 알려줍니다 (`doc/why-this-project-matters.md`의 7번 항목 참고).
- TDD 관점에서는 테스트가 있어야 "테스트하기 쉬운 구조"로 설계가 유도됩니다. 테스트를 나중에 붙이려고 하면 이미 결합도가 높아진 코드를 뜯어고쳐야 하는 경우가 많습니다.

즉 테스트의 가치는 버그를 잡는 것 자체보다, **변경할 때 드는 심리적/실질적 비용을 낮춰서 개발 속도를 유지**시켜준다는 데 있습니다.

## 2. 테스트 범위는 어디까지?

"모든 코드에 테스트가 들어가야 한다"는 이상론이지만, 실무에서는 항상 그 정도까지 필요하지는 않습니다. 기준을 코드 커버리지 숫자가 아니라 **"이 코드가 깨졌을 때 배포자가 눈치채지 못하면 어떤 일이 벌어지는가"**로 잡는 편이 현실적입니다.

- 도메인 규칙, 경계조건, 실패했을 때 비용이 큰 로직(인증, 결제, 데이터 정합성 등)은 테스트 우선순위가 높습니다.
- 프레임워크가 이미 검증해주는 부분(getter/setter, 단순 위임 코드)은 테스트 가치가 낮습니다.
- 이 프로젝트에서는 Fake 기반 소형 테스트와 `@SpringBootTest`/`@DataJpaTest` 기반 중형 테스트를 층위별로 나눠 쓰는데(`doc/why-this-project-matters.md`의 3번 항목), 이것도 "모든 걸 같은 방식으로 다 테스트하지 않는다"는 원칙의 실천입니다. 빠른 피드백이 필요한 로직은 Fake로, 실제 JPA 매핑·쿼리·HTTP 계층처럼 프레임워크와 맞닿는 부분은 중형 테스트로 검증합니다.

결국 "어디까지 테스트할 것인가"의 답은 "배포하는 사람이 확신을 가질 수 있는 최소한의 지점까지"이고, 그 지점은 코드마다 다릅니다.

## 3. 무엇을 테스트해야 하는가: Right-BICEP

Right-BICEP은 "무엇을 테스트 케이스로 만들어야 하는가"를 점검하는 체크리스트입니다.

- **Right** — 결과가 올바른가? 가장 기본적인 happy path 검증.
- **B**oundary — 경계조건은 맞는가? 흔히 놓치는 edge case들을 CORRECT 체크리스트로 세분화할 수 있습니다.
  - **C**onformance — 값이 기대하는 형식/포맷을 따르는가 (예: 이메일 형식).
  - **O**rdering — 순서가 있는 데이터가 올바른 순서로 나오는가/처리되는가.
  - **R**ange — 값이 유효한 최소/최대 범위 안에 있는가.
  - **R**eference — 외부에 대한 참조(다른 객체, 상태)가 올바른가. 메서드 바깥의 전제조건에 의존하지는 않는가.
  - **E**xistence — 값이 존재하는가(null, 빈 컬렉션, 0건 등).
  - **C**ardinality — 정확히 몇 개가 있어야 하는가(0개/1개/N개 문제, off-by-one).
  - **T**ime — 절대적/상대적 순서, 동시성, 타이밍 문제.
- **I**nverse relationship — 역관계로 교차 검증할 수 있는가 (예: 저장 후 조회해서 저장한 값과 같은지 확인).
- **C**ross-check — 다른 방법으로 같은 결과를 얻어 교차 검증할 수 있는가 (예: 다른 알고리즘, 이미 검증된 라이브러리와 비교).
- **E**rror conditions — 에러 상황을 강제로 유발했을 때 의도한 대로 처리되는가 (예외 발생, 실패 응답 등).
- **P**erformance — 응답 시간, 처리량 등 성능 특성이 기준을 만족하는가.

Right-BICEP은 "이 테스트 하나면 충분한가?"를 점검하는 용도로 쓰면 좋습니다. Happy path 하나만 짜고 넘어가기 쉬운데, 특히 B(경계조건)와 E(에러 조건)를 빠뜨리는 경우가 많습니다.

## 4. 실전 팁

- **ParameterizedTest** — 같은 검증 로직을 입력값만 바꿔 여러 번 돌리고 싶을 때 사용합니다. `if/else`로 나눠 매번 다른 테스트 메서드를 만드는 것보다 입력-기대값 쌍을 나열하는 편이 읽기 쉽고 케이스 추가도 쉽습니다.
- **assertAll** — 한 테스트 안에서 여러 검증을 할 때, 첫 번째 `assertThat`이 실패하면 나머지는 실행되지 않고 바로 끊깁니다. `assertAll`로 묶으면 모든 검증을 끝까지 실행하고 실패한 항목을 한 번에 보여줘서 디버깅 왕복이 줄어듭니다.
- **하나의 테스트는 하나만 검증한다, 단 케이스 바이 케이스** — "테스트 하나에 assert 하나"가 원칙이지만 기계적으로 지킬 필요는 없습니다. 핵심은 **표현력**입니다. 하나의 시나리오를 검증하는 데 필요한 assert들을 억지로 쪼개서 테스트명과 상황이 흩어지는 것보다, 하나의 의미 있는 단위로 묶어 읽었을 때 무엇을 검증하는지 바로 이해되는 쪽을 우선합니다.
- **Thread.sleep 대신 Thread.join, 그리고 가능하면 Awaitility** — 비동기/동시성 코드를 테스트할 때 `Thread.sleep(N)`으로 "이 정도 기다리면 끝나겠지"라고 추측하면 느리거나(너무 길게 잡으면) 불안정해집니다(너무 짧게 잡으면 flaky). 최소한 `Thread.join()`으로 스레드 종료를 명시적으로 기다리고, 조건이 만족될 때까지 폴링해야 하는 상황이라면 [Awaitility](https://github.com/awaitility/awaitility) 라이브러리를 쓰는 편이 훨씬 안정적이고 의도도 명확합니다 (`await().atMost(...).until(...)` 형태).
- **FIRST 원칙** — 좋은 테스트가 갖춰야 할 성질.
  - **F**ast — 테스트는 빨라야 한다. 느리면 자주 안 돌리게 되고, TDD의 빠른 피드백 루프가 무너집니다.
  - **I**ndependent — 테스트는 서로 독립적이어야 한다. 실행 순서나 다른 테스트의 부수효과에 의존하면 안 됩니다.
  - **R**epeatable — 몇 번을 돌려도 같은 결과가 나와야 한다. 랜덤 값, 현재 시각, 공유 상태에 의존하면 깨집니다(이 프로젝트에서 `ClockHolder`/`UuidHolder`를 포트로 분리하고 테스트에서 `TestClockHolder`/`TestUuidHolder`로 고정하는 이유가 이것입니다).
  - **S**elf-validating — 테스트 결과가 성공/실패로 자동 판정되어야 한다. 로그를 사람이 눈으로 읽고 판단해야 하면 테스트가 아닙니다.
  - **T**imely — 테스트는 적시에, 이상적으로는 구현 코드보다 먼저 작성되어야 한다. 나중에 몰아서 짜면 테스트하기 쉬운 구조로 설계를 유도하는 TDD의 효과를 놓칩니다.
